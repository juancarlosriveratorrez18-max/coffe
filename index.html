Aquí te explico ambos escenarios de forma clara. 1. Subir cambios de tu PC a GitHub (Push) Este es el flujo de trabajo más común. Ya hiciste cambios en tus archivos locales y quieres que se vean reflejados en la web. Verifica el estado: Abre tu terminal en la carpeta del proyecto y revisa qué archivos cambiaste. git status Prepara los archivos: Añade los cambios al área de preparación (staging). git add . (El punto añade todo, o puedes usar git add nombre_del_archivo). Crea un commit: Ponle una etiqueta a tu versión con un mensaje descriptivo. git commit -m "Descripción de los cambios realizados" Sube los cambios: Envía todo a la nube. git push origin main (Nota: Asegúrate de si tu rama principal se llama main o master). 2. Bajar cambios de GitHub a tu PC (Pull) Usa esto si editaste algo directamente en la web de GitHub o si un compañero de equipo subió código nuevo y necesitas tenerlo tú. Imagen de git pull workflow Shutterstock Sincroniza tu local: git pull origin main Tip de pro: Si te sale un error de "conflictos", significa que GitHub no sabe qué versión del código elegir porque ambos cambiaron la misma línea. Tendrás que abrir los archivos, elegir el código correcto y repetir el proceso de Push. 3. Actualizar un "Fork" (Sincronizar con el original) Si copiaste el repositorio de otra persona (hiciste un Fork) y el dueño original lo actualizó, sigue estos pasos: En la página de tu repositorio en GitHub, verás un botón que dice "Sync fork". Haz clic en "Update branch". Luego, en tu PC, haz un git pull para recibir esas mejoras.
